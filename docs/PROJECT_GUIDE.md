## 1. How the System Works (Big Picture)

The Rice DSS diagnoses rice paddy diseases through a **3-step process**:

### Step 1: Collect Input
The farmer provides one or both of:
- **Questionnaire answers** — 9 sections covering symptoms, weather, soil, fertilizer, field history
- **Leaf photo** — a close-up image of the affected rice leaf

### Step 2: Analyse
The system runs two independent analysis engines:
- **Scoring Engine** (rule-based) — 6 scoring functions evaluate answers against scientifically validated disease profiles. Each function produces a score from 0.0 to 1.0.
- **ML Classifier** (image-based) — An EfficientNetV2B0 CNN classifies the leaf photo into one of 4 classes (Bacterial Blight, Blast, Brown Spot, Healthy).

### Step 3: Decide and Recommend
The **Decision Engine** combines both sources through an 8-step priority hierarchy:

```
STEP 0  Validate all inputs (strip bad values, handle missing data)
STEP 1  Run all 6 questionnaire scoring functions
STEP 2  Check if evidence is too weak → "Uncertain" or "Out of Scope"
STEP 3  Non-biotic override: if iron toxicity / N deficiency / salt toxicity
        scores high → return immediately (ML cannot detect these)
STEP 4  Pathognomonic lock: if morning ooze is present and bacterial blight
        score >= 0.60 → lock to Bacterial Blight (near-unique biological marker)
STEP 5  Strong questionnaire dominance: if top Q score >= 0.80 → use it directly
        (if ML strongly disagrees → report ambiguity, don't silently override)
STEP 6  Safe ML fusion: blend Q and ML scores (60% Q + 40% ML) for biotic
        conditions only
STEP 7  Evaluate final biotic scores → return diagnosis or "Ambiguous" if
        two conditions are too close
```

The output includes: diagnosed condition, confidence level, all 6 scores, actionable recommendations (in English and Khmer), and any warnings.

```
rice_dss/
│
├── dss/           Core logic — scoring, decision, validation, recommendations
│                  This is the BRAIN of the system. All diagnosis logic lives here.
│                  These files are FROZEN — do not modify weights or thresholds.
│
├── api/           REST API — exposes the DSS as HTTP endpoints via FastAPI
│                  This is the INTERFACE for external clients (frontends, mobile apps).
│                  Contains no logic — just receives requests and delegates to dss/.

│                  This is the IMAGE ANALYSIS component.
│                  Trains a CNN classifier and bridges its output to the DSS format.
│
├── ui/            Streamlit demo — a local testing interface for the team
│                  This is a TESTING TOOL, not the production frontend.
│                  Imports DSS and ML directly (no API server needed).
│

│                  Validates every aspect of the system: DSS logic, API, ML, Grad-CAM, secondary conditions.

│                  Generated by ml/train.py and ml/evaluate.py.

├── data/          Training images — 9,200 leaf photos in 4 class folders
│                  Gitignored (too large for version control).
│

                   Gitignored. Generated by dss/logger.py at runtime.
```

## 3. `dss/` — Core DSS Engine

This is the most important directory. It contains all the diagnosis logic, and it is **frozen** — the scoring weights, thresholds, and decision hierarchy must not be modified without re-validating all 20 scientific test cases.

### `dss/__init__.py` — Package Exports

**What it does:** Makes `dss` a Python package and re-exports the most commonly used functions so other code can do:
```python
from dss import run_dss, generate_output, validate_answers
```

**Who calls it:** Everything — `api/main.py`, `ui/app.py`, `tests/`, `run_local.py`

---

### `dss/validation.py` — Input Validation

**What it does:** Takes raw farmer answers (a Python dict) and cleans them:
- Strips invalid values (e.g., if someone passes `growth_stage: "banana"`, it becomes `None`)
- Normalises lists and strings
- Handles missing fields safely (replaces with `None`, which scoring functions treat as neutral)
- Contains `VALID_*` sets — the exhaustive list of accepted values for every field
- Contains Khmer comments referencing the original PDF source materials

**Also contains:** `validate_ml_probabilities()` — checks that ML output has exactly the 3 required keys (`blast`, `brown_spot`, `bacterial_blight`), values are in [0.0, 1.0], and warns if the sum is not approximately 1.0.

**Who calls it:**
- `dss/decision.py` → `validate_answers()` is the first thing `generate_output()` calls (STEP 0)
- `api/main.py` → `validate_answers()` is called by the `/explain` endpoint
- `ui/app.py` → called before running the explainer

**Key design note:** Validation is intentionally lenient — it makes bad data neutral rather than throwing errors. This ensures the DSS always produces *some* output, even with partial or messy farmer input.

---

### `dss/scoring.py` — Scoring Engine (832 lines)

**What it does:** Contains 6 individual scoring functions, one per condition:
- `score_blast(answers)` — Rice Blast (fungal)
- `score_brown_spot(answers)` — Brown Spot (fungal)
- `score_bacterial_blight(answers)` — Bacterial Blight (bacterial)
- `score_iron_toxicity(answers)` — Iron Toxicity (non-biotic)
- `score_n_deficiency(answers)` — Nitrogen Deficiency (non-biotic)
- `score_salt_toxicity(answers)` — Salt Toxicity (non-biotic)

Each function follows the same pattern:
1. Start at `score = 0.0`
2. Check each relevant answer field — add points for supporting signals, subtract for contradicting ones
3. Multiply by `confidence_modifier` (0.65–1.0, based on farmer's self-reported confidence)
4. Clamp result to [0.0, 1.0] using `cap_score()`

**Also contains:**
- `compute_all_scores(answers)` — runs all 6 scoring functions and the two flag detectors, returns `{scores: {...}, flags: {sheath_blight, out_of_scope}}`
- `get_confidence_modifier(answers)` — returns the farmer confidence multiplier
- `flag_sheath_blight(answers)` — detects possible Sheath Blight (advisory only, outside system scope)
- `flag_out_of_scope(answers)` — detects when symptoms don't match any known pattern
- Helper functions: `cap_score()`, `has_symptom()`, `has_location()`, `has_additional()`

**Who calls it:** `dss/decision.py` → `compute_all_scores()` is called in STEP 1 of `generate_output()`

**Key design note:** All scoring weights are derived from agricultural literature (PDF1 = CARDI disease handbook, PDF2 = soil/fertilizer management guide). Inline comments reference these sources.

---

### `dss/decision.py` — Decision Hierarchy (IMMUTABLE)

**What it does:** Contains `generate_output()` — the single most important function in the system. It implements the 8-step controlled dominance hierarchy described above.

**How it works (step by step):**
1. Validates input via `validate_answers()`
2. Runs all 6 scoring functions via `compute_all_scores()`
3. Checks if evidence is too weak (out-of-scope or uncertain)
4. Checks non-biotic conditions — if strong, returns immediately (overrides ML)
5. Checks for pathognomonic lock (morning ooze → Bacterial Blight)
6. Checks for strong questionnaire dominance (>= 0.80)
7. If ML probabilities are available, fuses Q and ML scores (60/40 blend)
8. Evaluates final biotic scores and returns the result

**Who calls it:** `dss/mode_layer.py` → both `questionnaire_only_logic()` and `hybrid_logic()` delegate to `generate_output()`

**Key design note:** This function is IMMUTABLE. Changing any step, threshold, or condition ordering breaks the scientifically validated test cases.

---

### `dss/output_builder.py` — Output Formatting + Constants

**What it does:** Two responsibilities:
1. **Constants** — All threshold values, condition sets, and display labels are defined here:
   - `THRESHOLD_PROBABLE = 0.65` — score needed for "Probable" diagnosis
   - `THRESHOLD_POSSIBLE = 0.40` — score needed for "Possible" diagnosis
   - `THRESHOLD_AMBIGUOUS_GAP = 0.20` — minimum gap between top two scores
   - `QUESTIONNAIRE_WEIGHT = 0.60` / `ML_WEIGHT = 0.40` — hybrid fusion weights
   - `CONDITION_LABELS` — bilingual display names (English + Khmer)
   - `CONFIDENCE_LABELS` — bilingual confidence level names

2. **Output builders** — Functions that format the final output dictionary:
   - `build_standard_output()` — when a condition is confidently identified (includes `secondary_conditions`)
   - `build_ambiguous_output()` — when two conditions conflict (returns empty `secondary_conditions`)
   - `build_uncertain_output()` — when evidence is too weak (returns empty `secondary_conditions`)
   - `build_out_of_scope_output()` — when symptoms are outside system scope (returns empty `secondary_conditions`)
   - `append_sheath_warning_if_needed()` — injects Sheath Blight advisory warning
   - `determine_confidence_label()` — converts score + gap to confidence level string

3. **Secondary conditions extraction** — `_extract_secondary_conditions(all_scores, primary_key)` surfaces conditions scoring >= `THRESHOLD_POSSIBLE` (0.40) that aren't the primary diagnosis. Returns a list of `{condition, condition_key, score}` sorted by score descending. This catches clinically relevant co-occurrences (e.g., Brown Spot + N Deficiency) from scores the DSS already computes.

**Who calls it:** `dss/decision.py` exclusively — every branch of `generate_output()` calls one of these builders.

**Key design note:** The output dictionary structure is the contract with `api/schemas.py`. If you add or remove keys here, you must update the Pydantic schemas too.

---

### `dss/recommendations.py` — Treatment Recommendations

**What it does:** Returns condition-specific, farmer-facing advice structured as:
```python
{
    'immediate':  [...],   # Actions to take right now
    'preventive': [...],   # Long-term prevention measures
    'monitoring': '...',   # What to watch for
    'consult':    True/False  # Whether to seek expert help
}
```

All recommendation text is sourced from PDF1 (CARDI disease handbook) and PDF2 (soil/fertilizer guide). Some recommendations are **personalised** based on the farmer's answers:
- Blast at flowering stage → adds urgency warning
- Brown spot + no fertilizer applied → emphasises nitrogen application
- Iron toxicity at tillering → notes that draining now is optimal timing
- N deficiency → recommends fertilizer rates specific to the farmer's soil type
- Salt toxicity + excessive fertilizer → identifies cause

**Who calls it:** `dss/output_builder.py` → `build_standard_output()` calls `get_recommendations()`

---

### `dss/mode_layer.py` — 3-Mode Routing

**What it does:** The unified entry point for all DSS operations. `run_dss()` takes a mode parameter and routes to the appropriate logic:

| Mode | Function Called | What Happens |
|------|----------------|--------------|
| `"questionnaire"` | `questionnaire_only_logic()` | Sets `ml_probabilities = None`, then calls `generate_output()` |
| `"ml"` | `ml_only_logic()` | Uses ML probabilities only, cannot detect non-biotic stresses |
| `"hybrid"` | `hybrid_logic()` | Passes everything to `generate_output()` (recommended mode) |

**ML-only mode** is unique — it does NOT use `generate_output()` at all. Instead, it directly reads the ML probabilities, applies its own thresholds (`ML_THRESHOLD_STRONG = 0.65`, `ML_THRESHOLD_POSSIBLE = 0.40`), and builds a result. It always injects a warning that non-biotic stresses cannot be detected. All ML-only return paths include `'secondary_conditions': []` for schema consistency (secondary conditions are not meaningful without questionnaire scores).

Every mode also logs the run via `dss_logger.log_run()` and appends a `mode_used` field to the output.

**Who calls it:**
- `api/main.py` → all 5 diagnosis endpoints delegate to `run_dss()`
- `ui/app.py` → calls `run_dss()` directly for in-process diagnosis

---

### `dss/explainer.py` — Score Traceability

**What it does:** Mirrors every scoring function in `scoring.py` but instead of computing scores, it records each signal as a human-readable explanation:
```python
{
    'field': 'weather',
    'value': 'high_humidity',
    'effect': '+',
    'weight': 0.30,
    'reason': 'High humidity essential for blast (PDF1)'
}
```

**Why it exists:**
1. FYP defence — proves the system's reasoning to evaluators
2. Farmer transparency — shows *why* the system made its diagnosis
3. Debugging — identifies which signals are contributing most
4. Explainability is a graded criterion in the FYP evaluation

**Who calls it:**
- `api/main.py` → the `/explain` endpoint
- `ui/app.py` → the explanation panel in the Streamlit UI

**Key design note:** This module is purely observational — it does NOT modify any scores. The actual scoring in `scoring.py` remains untouched.

---

### `dss/logger.py` — Audit Trail

**What it does:** Records every DSS run as a structured log entry containing: timestamp, input summary, all 6 scores, final condition, confidence level, and mode used.

**Storage:** In-memory list (for session analysis) + optional JSONL file at `logs/dss_runs.jsonl`.

**Features:**
- `log_run()` — records a single DSS run
- `get_runs()` — returns all logged runs from this session
- `get_summary()` — returns aggregate statistics (status/condition/confidence distributions)
- `export_csv()` — exports to CSV for analysis in Excel/Sheets

**Who calls it:** `dss/mode_layer.py` → `run_dss()` logs every invocation. Also used by `api/main.py` (log endpoints) and `ui/app.py` (session log sidebar).

**Key design note:** This module has zero impact on DSS logic. Removing it would not change any diagnosis.

## 4. `api/` — REST API

The API is a thin layer that exposes the DSS over HTTP. It contains **no diagnosis logic** — it only receives requests, validates them with Pydantic, delegates to `run_dss()`, and returns the result.

### `api/__init__.py` — Package Marker
Empty file that makes `api` a Python package.

### `api/main.py` — FastAPI Application (10 Endpoints)

**Live deployment:** `https://rice-dss-137747818788.asia-southeast1.run.app`

**Endpoints:**

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/` | GET | API root — lists all available endpoints |
| `/questionnaire` | POST | Questionnaire-only diagnosis |
| `/ml-only` | POST | ML probabilities only |
| `/hybrid` | POST | Full hybrid mode (recommended) |
| `/predict-image` | POST | Upload a leaf photo → ML diagnosis |
| `/hybrid-image` | POST | Upload photo + questionnaire JSON |
| `/explain` | POST | Signal-level score breakdown |
| `/logs/summary` | GET | Aggregated run statistics |
| `/logs/runs` | GET | Recent run history |
| `/health` | GET | System status + model availability |

**Key features:**
- **CORS** — reads `CORS_ORIGINS` environment variable (comma-separated), defaults to `*` for development
- **Image validation** — checks MIME type (JPEG/PNG/WebP only) and file size (10 MB max) on upload endpoints
- **Lazy model loading** — the ML model is loaded on first request via `get_inference_model()` singleton, avoiding TensorFlow import at startup. Returns `None` (HTTP 503) if no model file exists.
- **Root endpoint** — returns a JSON overview of the service and all available endpoints for discoverability

**How the three JSON endpoints work (same pattern):**
1. Receive `QuestionnaireRequest` payload (Pydantic validates the structure)
2. Convert to plain dict via `_request_to_dict()`
3. Call `run_dss(raw, mode=...)` with the appropriate mode
4. Return the dict (FastAPI serialises it against `DSSResponse`)

**How image endpoints work:**
1. Receive uploaded file via `UploadFile`
2. Validate MIME type and file size
3. Load ML model via `get_inference_model()`
4. Call `model.predict_from_bytes()` → returns 3-key probability dict
5. For `/predict-image`: run `run_dss(mode="ml")` with those probabilities
6. For `/hybrid-image`: parse questionnaire JSON from Form field, inject ML probabilities, run `run_dss(mode="hybrid")`

### `api/schemas.py` — Pydantic Request/Response Models

**What it does:** Defines the data contracts between the API and any client:

- `QuestionnaireRequest` — incoming payload with all questionnaire fields + `ml_probabilities`. All fields are Optional (the DSS handles missing values safely).
- `DSSResponse` — standard output format with condition, score, confidence, recommendations, warnings.
- `ImagePredictionResponse` — extends `DSSResponse` with `ml_probabilities` field for `/predict-image`.
- `ExplainResponse` — output from `/explain` endpoint.
- `SignalEntry`, `ConditionExplanation` — nested models for explanation data.

**Key design note:** Field names in `QuestionnaireRequest` MUST match the keys expected by `validate_answers()` in `dss/validation.py`. The `DSSResponse` mirrors the output of `generate_output()`.

## 5. `ml/` — Machine Learning Pipeline

### `ml/__init__.py` — Package Marker
Empty file that makes `ml` a Python package.

### `ml/dataset.py` — Dataset Loader

**What it does:** Loads rice disease images from a directory structure for training:

```
data/
├── bacterial_blight/   (2,300 images)
├── blast/              (2,300 images)
├── brown_spot/         (2,300 images)
└── healthy/            (2,300 images)
```

**Key constants:**
- `CLASS_NAMES = ['bacterial_blight', 'blast', 'brown_spot']` — the 3-class DSS contract
- `ALL_CLASS_NAMES = ['bacterial_blight', 'blast', 'brown_spot', 'healthy']` — the 4-class training set
- `DEFAULT_IMG_SIZE = 224` — standard input dimensions for both MobileNetV2 and EfficientNetV2B0

**Also provides:**
- `load_dataset()` — uses `tf.keras.utils.image_dataset_from_directory()` to create training/validation splits (80/20), with caching and prefetching for performance
- `build_augmentation_layer()` — data augmentation (flips, rotation, zoom, brightness, contrast) designed for field photography conditions
- `get_class_index_map()` — maps model output indices to DSS condition keys

### `ml/train.py` — Training Pipeline

**What it does:** Trains a CNN image classifier with configurable backbone architecture. Supports MobileNetV2 and EfficientNetV2B0 via `--backbone` flag. Training uses two phases:

1. **Phase 1 (backbone frozen):** Only the classification head is trained. The backbone (pretrained on ImageNet) is kept frozen to use its learned feature representations.
2. **Phase 2 (fine-tuning, optional):** The top N backbone layers are unfrozen and trained at a lower learning rate (1e-5). Can be disabled with `--fine_tune_epochs 0`.

**Production model architecture (EfficientNetV2B0):**
```
Input (224x224x3)
  → Data Augmentation (random flip, rotation, zoom, brightness, contrast, translation)
  → EfficientNetV2B0 Preprocessing
  → EfficientNetV2B0 Backbone (pretrained, frozen)
  → Global Average Pooling
  → Dense(256, relu) + Dropout(0.3)
  → Dense(4, softmax)  ← 4 classes including healthy
```

**Callbacks:**
- `ModelCheckpoint` — saves best model by validation accuracy
- `EarlyStopping` — stops if validation loss plateaus (patience=5)
- `ReduceLROnPlateau` — halves learning rate if loss plateaus (patience=3)

**Output files:**
- `models/rice_disease_model.keras` — the trained model weights
- `models/rice_disease_model.meta.json` — class names + image size metadata
- `models/evaluation/training_history.json` — epoch-by-epoch metrics
- `models/evaluation/training_history.png` — accuracy and loss curves

**How to run:**
```bash
python -m ml.train --data_dir data/ --epochs 20
```

### `ml/inference.py` — Inference Wrapper (4-to-3 Bridging)

**What it does:** Loads the trained 4-class model and converts its output to the 3-key probability dict that the DSS expects.

**The bridging problem:** The model trains on 4 classes (including "healthy") because having a healthy class improves feature learning. But the DSS only knows about 3 biotic diseases. The `_bridge_to_dss()` method solves this:

- **If healthy probability >= 60%:** The model thinks the leaf is healthy. Return uniform probabilities (1/3 each) so the DSS ignores ML and relies on questionnaire evidence only.
- **Otherwise:** Drop the healthy class, take the 3 disease probabilities, and renormalise them to sum to 1.0.

**Class: `RiceDSSInference`**
- `predict_from_image(path)` — loads an image file and returns 3-key probabilities
- `predict_from_bytes(bytes)` — processes raw bytes (from HTTP upload) and returns 3-key probabilities
- `get_raw_probabilities(path)` — returns all 4 raw class probabilities (for debugging)

**Metadata loading:** On init, checks for `models/rice_disease_model.meta.json` to load the correct class names and image size. Falls back to defaults if the sidecar file is missing.

**Who uses it:**
- `api/main.py` → `get_inference_model()` creates a singleton `RiceDSSInference` instance for the image endpoints
- `ui/app.py` → `load_ml_model()` creates a cached instance for the Streamlit UI

### `ml/evaluate.py` — Model Evaluation

**What it does:** Evaluates the trained model on the validation set and generates:
- Classification report (precision, recall, F1 per class) → saved as `models/evaluation/classification_report.txt`
- Confusion matrix (visual) → saved as `models/evaluation/confusion_matrix.png`
- Overall accuracy metric

**How to run:**
```bash
python -m ml.evaluate --model_path models/rice_disease_model.keras --data_dir data/
```

## 6. `translations/` — Language Layer

The translations module is a **post-processing layer** that sits between the frozen DSS output and the API/UI response. It enables bilingual support (English + Khmer) without modifying any frozen DSS files.

### `translations/__init__.py` — Package Exports

Exports: `translate_output`, `get_ui_labels`, `get_label_map`

### `translations/core.py` — Translation Logic

**What it does:** The central translation engine:
- `translate_output(output, lang)` — translates a DSS output dict to the target language. Parses bilingual strings (e.g., "ជំងឺប្លាស (Rice Blast)") and extracts the correct language.
- `split_bilingual(label)` — splits "ខ្មែរ (English)" format strings into `(khmer, english)` tuples
- `_translate_recommendations(recs, lang, condition_key, answers)` — for English, applies string replacements from `RECOMMENDATIONS_EN_REPLACEMENTS` to improve frozen DSS text (removes banned chemicals, simplifies academic phrasing); for Khmer, uses static `RECOMMENDATIONS_KM` override
- `_translate_secondary_conditions(conditions, lang)` — translates bilingual condition names in the `secondary_conditions` list
- `_translate_ambiguous_between(items, lang)` — translates bilingual labels in the `ambiguous_between` list
- `get_ui_labels(lang)` — returns the full UI label dictionary for the given language
- `get_label_map(base_map, lang)` — extracts monolingual values from bilingual label dicts (used by UI dropdowns)

**Who calls it:**
- `api/main.py` → calls `translate_output()` on every DSS response before returning
- `ui/app.py` → calls `get_ui_labels()` and `get_label_map()` for label translation

### `translations/en.py` — English Labels + Recommendation Refinements

**What it does:**
- `UI_LABELS_EN` — English UI strings (section headers, button labels, tooltips, `also_consider`, `fertilizer_disclaimer`, etc.)
- `RECOMMENDATIONS_EN_REPLACEMENTS` — maps frozen DSS recommendation strings to improved farmer-friendly versions (removes Benlate (banned), removes PDF references, simplifies academic phrasing, softens soil-specific N rates with "approximately" framing)
- `RECOMMENDATIONS_EN_MONITORING` — maps frozen monitoring strings to improved versions
- `RECOMMENDATIONS_EN_EXTRAS` — appends extra items per condition (e.g., "no effective chemical cure" for bacterial blight)

### `translations/km.py` — Khmer Translations

**What it does:**
- `UI_LABELS_KM` — Khmer UI strings (mirrors `UI_LABELS_EN` structure)
- `RECOMMENDATIONS_KM` — full Khmer recommendation overrides per condition (since Khmer text is a complete rewrite, not string replacements)
- `WARNINGS_KM` — Khmer warning message translations

**Key design note:** The translation layer is the only way to modify recommendation text or add language support without touching the frozen `dss/recommendations.py`. English refinements use targeted string replacements to preserve the DSS's dynamic personalisation (e.g., soil-specific rates). Khmer uses full static overrides because the translation is a complete rewrite.

## 7. `ui/` — Streamlit Demo Interface

### `ui/app.py` — Testing UI

**What it does:** A local web interface for testing the DSS. Supports all 3 diagnosis modes:
- **Questionnaire Only** — shows the questionnaire form (Quick or Detailed)
- **Image Only (ML)** — shows image upload only
- **Hybrid (Recommended)** — shows both questionnaire and image upload

**Smart Questionnaire** — The questionnaire has two depth modes:
- **Quick mode** (~6 key questions): growth stage, symptoms, symptom location, farmer confidence, fertilizer, additional symptoms
- **Detailed mode** (all fields): adds weather, water condition, timing, spread pattern, onset speed, field history, soil type, with progressive disclosure (conditional fields appear based on previous answers)
- A **diagnostic strength indicator** shows Limited/Moderate/Strong based on how many fields are filled
- The test case loader auto-switches to Detailed mode

**Key features:**
- **Bilingual labels** — Every dropdown option shows both English and Khmer text
- **Language toggle** — English / ខ្មែរ switch at the top of the page
- **Test case loader** (sidebar) — Load any of the 20 validated test cases to pre-fill the form
- **Score breakdown** — Visual bar charts showing all 6 condition scores
- **Secondary conditions** — "Also Consider" section shows co-occurring conditions scoring >= 0.40
- **Fertilizer disclaimer** — For N deficiency cases, shows a note that rates are guidelines to adjust for local conditions
- **Trust messaging** — Non-biotic condition disclaimer + confidence caveat
- **Explanation panel** — Detailed signal-by-signal breakdown (why the system made this diagnosis)
- **ML probability bars** — Shows raw ML model output in Hybrid mode
- **Grad-CAM** — Side-by-side original image + heatmap overlay
- **Debug panels** — Expandable JSON views of raw output, raw answers, and full explanation

**How it works (in-process, no API needed):**
- Imports `dss.mode_layer.run_dss` directly — no HTTP calls to the API server
- Imports `ml.inference.RiceDSSInference` directly — model loaded via `@st.cache_resource` (Streamlit singleton, loaded once, reused across browser sessions)

**Label maps:** The file defines ~15 dictionaries (e.g., `GROWTH_STAGE_LABELS`, `SYMPTOM_LABELS`) that map raw DSS keys to human-readable bilingual labels. These are used by the `labeled_select()` and `labeled_multiselect()` helper functions.

**How to run:**
```bash
source .venv312/bin/activate && streamlit run ui/app.py
```

## 8. `tests/` — Test Suite

142 tests across 7 files, all passing (some may be skipped depending on optional TF/Plotly dependencies).

### `tests/test_dss.py` — Core Disease Cases (30 tests)

**What it does:** Contains `TEST_CASES` — a list of 20 scientifically validated scenarios, each specifying farmer answers and the expected DSS output. These are the "ground truth" for the system.

**Example cases:**
- Case 01: Classic Blast (Flowering Stage) → expected: `blast`
- Case 03: Bacterial Blight with morning ooze → expected: `bacterial_blight` (pathognomonic lock)
- Case 06: Iron Toxicity → expected: `iron_toxicity`
- Case 11: Insufficient data → expected: `uncertain`
- Case 14: Sheath Blight → expected: sheath blight warning present

Also tests validation behaviour, output structure, and confidence labelling.

### `tests/test_hybrid.py` — Hybrid ML Fusion (25 tests)

**What it does:** Tests the interaction between questionnaire scoring and ML probabilities:
- ML and questionnaire agree → fused output
- ML and questionnaire disagree → ambiguous output
- Non-biotic condition overrides ML even when ML says 0.95 for a biotic disease
- Missing ML probabilities → falls back to questionnaire
- Pathognomonic lock still works in hybrid mode

### `tests/test_robustness.py` — Adversarial Inputs (14 tests)

**What it does:** Tests the system's resilience to bad or unusual inputs:
- Noise injection (random invalid values added to answers)
- Empty answers (all fields missing)
- Contradictory signals (e.g., both dry and flooded)
- Invalid types (numbers where strings expected)
- Ensures the system always returns a valid output dict, never crashes

### `tests/test_api.py` — API Endpoints (14 tests)

**What it does:** Tests all 9 API endpoints using FastAPI's `TestClient`:
- Valid requests return correct structure
- Invalid ML probabilities are handled gracefully
- Image upload endpoints work (with mock model)
- Health endpoint reports model status
- CORS headers are present

### `tests/test_ml.py` — ML Pipeline (35 tests)

**What it does:** Tests the ML components without requiring TensorFlow:
- Dataset constants (`CLASS_NAMES`, `ALL_CLASS_NAMES`)
- Inference bridging (4-class to 3-class conversion)
- Healthy class threshold (exactly at 0.60 boundary)
- Placeholder probabilities
- Metadata loading
- Multi-architecture backbone selection
- Experiment tracking directory structure

### `tests/test_gradcam.py` — Grad-CAM Explainability (10 tests)

**What it does:** Tests the Grad-CAM visual explainability module:
- Heatmap generation and overlay
- Last convolutional layer detection
- PIL Image output validation
- Graceful fallback when model has no conv layers
- API schema validation (gradcam_base64 field)

### `tests/test_secondary.py` — Secondary Conditions (12 tests)

**What it does:** Tests the secondary conditions extraction and translation:
- Brown Spot + N Deficiency co-occurrence (Case 8 pattern)
- Clean blast case produces no secondary conditions
- Primary condition is never included in secondary list
- Secondary conditions are sorted by score descending
- All secondary scores are >= `THRESHOLD_POSSIBLE` (0.40)
- Entry structure validation (`condition`, `condition_key`, `score`)
- Uncertain/out-of-scope/ML-only outputs return empty `secondary_conditions`
- English and Khmer translation of secondary condition names
- Full pipeline tests via `run_dss()` in questionnaire and hybrid modes

**How it works:** Cloud Run builds the Docker image from source, pushes it to Artifact Registry, and serves it as a managed container. Cold starts take ~15-20s (TensorFlow loading); subsequent requests are fast.

**To redeploy after changes:**
```bash
gcloud run deploy rice-dss --source . --platform managed --region asia-southeast1 --memory 1Gi --allow-unauthenticated --set-env-vars "CORS_ORIGINS=*"
```

### `Dockerfile`
Builds a container image based on `python:3.12-slim` with TensorFlow CPU. Copies all project code (excluding data, venvs, caches via `.dockerignore`). Default command starts the API server on port 8000.

### `docker-compose.yml`
Defines two services:
- **api** — Always runs. Builds from Dockerfile, exposes port 8000, volume-mounts `models/` and `logs/`, health-checked via `/health` endpoint.
- **ui** — Only runs with `--profile demo`. Starts Streamlit on port 8501, depends on `api` service.

### `.dockerignore`
Excludes from Docker build: `.venv*`, `data/`, `__pycache__/`, `.git/`, `*.pyc`, `.DS_Store`, `*.md` (except `requirements.txt`).

## 12. Data Flow: End-to-End Walkthrough

### Questionnaire-Only (via API)

```
Client (POST /questionnaire)
  → api/main.py: questionnaire_endpoint()
    → _request_to_dict(): Pydantic model → plain dict
    → dss/mode_layer.py: run_dss(raw, mode="questionnaire")
      → questionnaire_only_logic(): sets ml_probabilities = None
        → dss/decision.py: generate_output(answers)
          → STEP 0: dss/validation.py: validate_answers()
          → STEP 1: dss/scoring.py: compute_all_scores()
          → STEPs 2-7: decision hierarchy
          → dss/output_builder.py: build_*_output()
            → dss/recommendations.py: get_recommendations()
      → dss/logger.py: dss_logger.log_run()
  ← JSON response (DSSResponse)
```

### Image Upload (via API)

```
Client (POST /predict-image, multipart upload)
  → api/main.py: predict_image()
    → Validate MIME type + file size
    → get_inference_model(): loads ML model (singleton)
    → ml/inference.py: model.predict_from_bytes(image_bytes)
      → preprocess_image_bytes(): decode, resize, expand dims
      → model.predict(): raw 4-class softmax
      → _bridge_to_dss(): 4→3 class conversion
    → dss/mode_layer.py: run_dss({'ml_probabilities': probs}, mode="ml")
      → ml_only_logic(): threshold check, build result
  ← JSON response (ImagePredictionResponse)
```

### Hybrid Mode (via Streamlit UI)

```
User fills questionnaire + uploads image in Streamlit
  → ui/app.py: builds raw_answers dict
    → load_ml_model(): cached ML model (singleton)
    → ml/inference.py: model.predict_from_bytes(image_bytes)
    → raw_answers['ml_probabilities'] = ml_probs
    → dss/mode_layer.py: run_dss(raw_answers, mode="hybrid")
      → hybrid_logic()
        → dss/decision.py: generate_output(raw_answers)
          → STEP 6: Safe ML fusion (60% Q + 40% ML)
    → dss/explainer.py: explain_scores(validated_answers)
    → Render result + explanation + ML bars
```

## 13. Key Relationships Between Files

### Constants Flow
```
dss/output_builder.py (defines thresholds, weights, condition sets, labels)
  ↓ imported by
dss/decision.py (uses thresholds in the 8-step hierarchy)
  ↓ called by
dss/mode_layer.py (routes to correct logic based on mode)
  ↓ called by
api/main.py + ui/app.py (entry points for users)
```

### ML Pipeline Flow
```
ml/dataset.py (CLASS_NAMES, ALL_CLASS_NAMES, load_dataset)
  ↓ imported by
ml/train.py (trains model, saves .keras + .meta.json)
  ↓ model file loaded by
ml/inference.py (RiceDSSInference, bridges 4→3 classes)
  ↓ called by
api/main.py (image endpoints) + ui/app.py (direct import)
```

### Validation Contract
```
dss/validation.py → VALID_* sets define accepted values
api/schemas.py → QuestionnaireRequest field names MUST match validation.py keys
ml/dataset.py → CLASS_NAMES MUST match BIOTIC_CONDITIONS in output_builder.py
```

### Output Contract
```
dss/output_builder.py → build_*_output() defines output dict structure
api/schemas.py → DSSResponse mirrors that structure exactly
```

## 14. Frozen Logic — What You Must Not Change

| What | Where | Why |
|------|-------|-----|
| Scoring weights | `dss/scoring.py` | Derived from agricultural literature (PDF1/PDF2) |
| Decision hierarchy (8 steps) | `dss/decision.py` | Scientifically validated order of operations |
| Threshold constants | `dss/output_builder.py` | Calibrated against 20 test cases |
| Fusion weights (60/40) | `dss/output_builder.py` | Ensures questionnaire dominance over ML |
| Condition sets | `dss/output_builder.py` | Non-biotic vs biotic determines override behaviour |
| Khmer comments | `dss/validation.py`, `dss/scoring.py` | Reference PDF sources, used in evaluation |
| Test case expected outputs | `tests/test_dss.py` | Ground truth from agricultural experts |
| Recommendation text | `dss/recommendations.py` | Sourced from validated PDF materials |

**Safe to modify:**
- API endpoints, schemas, and route handlers (`api/`)
- Translation layer — UI labels, recommendation refinements, new languages (`translations/`)
- UI layout, styling, and label maps (`ui/`)
- ML training hyperparameters (`ml/train.py`)
- Test assertions for new features (`tests/`)
- Infrastructure files (`Dockerfile`, CI, docker-compose)
- Documentation (`*.md`)
